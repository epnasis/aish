# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`aish` is a terminal AI agent (local Ollama model by default, cloud backends optional) that runs shell commands, edits files, and browses the web — with a mandatory user-approval gate on anything that mutates state. The README covers user-facing behavior thoroughly; this file covers what you need to change the code safely.

## Commands

```sh
uv run pytest                          # full test suite — no model/network needed
uv run pytest tests/test_agent.py      # one file
uv run pytest tests/test_agent.py -k compaction   # one test by keyword
uv run ruff check .                    # lint (also formats config: line-length 100)
uv run mypy                            # type check (CI-gated; config in pyproject.toml)
uv run aish                            # run from source
```

After changing code, the user's installed `aish` does NOT pick it up — uv freezes the wheel and caches it at the same version. Reinstall with:

```sh
uv tool install --force --reinstall --no-cache /path/to/aish
```

## Workflow: when to use a worktree

Before editing code, check `git status`. Work in a git worktree on a feature branch (via EnterWorktree) instead of this checkout when either:

- the change is **non-trivial** — a multi-file feature or refactor, not a small fix; or
- the tree is **not clean with changes you didn't make** — the user or another session is mid-work in this checkout.

Merge back to main after tests pass, then remove the worktree. Trivial fixes on a clean tree go directly in this checkout. The reinstall/ship step (`uv tool install ...`) always runs from this main checkout after merge — never from a worktree path.

## Architecture

The core invariant: **the model never executes anything directly.** Backends only return structured tool-call requests; `Agent._dispatch()` in `agent.py` is the single execution point, and `run_command` is unreachable there unless the `approve()` callback returns the (possibly user-edited) command.

Data flow: `cli.py` (REPL/argv, slash commands, rendering) constructs an `Agent` with a chat callable from `backends.py` and an `approve` callback, then calls `agent.run_task()`. The agent loops: model proposes tool calls → `_dispatch()` executes them (gated) → results appended → repeat until a final answer. Approver verdicts: `True`/edited-string approve, `None`/`False` deny, `Blocked(reason)` (denylist — unapprovable), `Denied(comment)` and `Approved(comment, command=None)` — both web-card-only (CLI approvers still return bool/str/None), and both constructed ONLY when the user typed a comment. **A comment makes approve and deny mean opposite things (issue #81):** `Denied(comment)` = STOP — the action is denied, the comment appended to the denial result, and the **stop gate** armed; `Approved(comment, ...)` = CONTINUE but ADJUST — the original action is HELD (never run, even if `command` was edited), and the model is told to rework it to the comment and re-propose (the adjusted action is approved again before it runs). The **stop gate** (`_stop_gate`, arms via `_arm_stop_gate`, only denials arm it): eager models run another tool before addressing a denial, so while `self._pending_comment_response` is set every tool call is refused; the main loop clears it only on a **text-only turn** (not a same-turn text+tool: chatty preamble alongside a command would otherwise slip past), and that text-only turn ends the task — deny means stop. `run_task` resets the flag so it never leaks across tasks. Approvals never arm the gate (continue); the HOLD + re-approval is what keeps an unwanted original command from running. The step budget is **progress-gated (#108)**, not a flat cap: a step counts as progress when a tool call yields a (tool, args, result) tuple seen for the first time (reusing the loop detector's `repeats` dict — no extra model call, no timer). A progressing task may run past `max_steps` up to the unconditional hard ceiling `max(max_steps, HARD_STEP_CEILING=60)`; `MAX_STALL_STEPS` (8) consecutive no-progress steps stop it early. The loop ends three ways: a text-only answer, a cancel, or `_finish_stopped()` — reached at a stall, the hard ceiling, or when loop detection fires (identical call + identical result: warn at 3 repeats, stop at 5) — which runs one final no-tools turn so the model reports completion state instead of cutting off; the ceiling is never silently exceeded.

- **`backends.py`** — routes `--model` strings to a chat callable. Every backend (Ollama, Gemini, Claude API, OpenAI) is adapted to the *exact* `ollama.chat` calling convention (`chat(model, messages, tools, options, think, stream)`), so `agent.py` never knows which provider it's on. New backends must preserve this shape.
- **`claude_max.py`** — the exception: routes through the Claude Agent SDK / local `claude` CLI login. It strips Claude Code to bare inference and injects aish's own tools, so the approval gate still applies. Keeps its own session state.
- **`approval.py`** — conservative parser classifying commands as read-only for auto-approval. Philosophy: only approve what it *positively* understands; anything ambiguous falls through to a prompt. Auto-approval is scoped to session roots (launch dir + `/add-dir` + directories the user trusted at a prompt); path arguments escaping them (absolute, `~`, `..`, symlinks resolved) prompt even when allowlisted — `escaping_dirs()` names the escapes so the prompt/card can offer "trust this directory" (`Agent.trust_root`, session-memory only). When touching this file, err toward prompting.

Model execution is **stateless**: every `run_command` runs in the project directory (`Agent.cwd`), a bare model-issued `cd` is rejected with guidance instead of executing (see `CD_NOT_STICKY` in `agent.py`), and excursions are `cd x && ...` subshells (`cd` is in `SAFE_COMMANDS` because it is subshell-scoped and root-checked). Only user actions move cwd: `/cd` and its alias `!cd` both move cwd AND re-anchor `roots[0]` (`Agent.rebase`). This keeps the model permanently anchored to the project directory — do not reintroduce model-driven cwd mutation.
- **`tools.py`** — `run_command` (approval-gated shell) and `read_docs` (auto-approved, so it takes only a bare command name validated against PATH — never a shell string). The denylist (unrecoverable commands blocked even with approval) lives here.
- **`files.py`** — pure `plan_*` functions compute (old, new, diff) without touching disk; `commit()` writes. The gap between plan and commit is where the diff is shown and approval obtained — nothing is written unseen.
- **`web.py`** — `web_search`/`read_url`, auto-approved but their inputs leave the machine, so every call is echoed and fetched content is wrapped in an untrusted-content banner. http/https only.
- **`session.py`** — append-only JSONL per session in `~/.local/state/aish/`: conversation (for `--resume`), audit trail of every command decision, plus `kind:"trace"` records — the structured activity-trace steps (the same dicts the web renders), logged from BOTH entry points via the agent's `step_log` sink so the trace is UI-agnostic and survives eviction/restart. Terminal-block framing (`kind:"cmd_start"`/`"cmd_end"`) is persisted the same way via the `command_log` sink. `reconstruct_events()` replays a logged session as the EXACT event stream a rich client emits live — `user`/`step`/`done` plus, for each run_command, its full `command_start → stream → command_end → tool` sequence — so a cold-loaded web session feeds the frontend identical events and renders an identical terminal block (no fallback path). The command's output rides on the `tool` step (not duplicated in the framing records) and is spliced back in as one `stream` with run_command's trailing `[exit code: N]` marker stripped (`_EXIT_MARKER_RE`), matching the live body where the code arrives via `command_end`; a legacy tool step with no framing gets a synthesized block. One `done` per task carries the final assistant text; it returns `None` for pre-trace logs, and the server falls back to a flat `history` blob then. `step_log`/`command_log` are orthogonal to the agent's `on_step`/`on_command_*` (rich-renderer hooks; `on_step` gates terminal chatter via `_note`) — the CLI wires the log sinks only, so it logs without changing its inline output. Its ranked excerpt search (`search_excerpts`, output hard-capped) backs the sessions section of the `recall` tool; the agent gets `state_dir` + `current_session` at construction and excludes the live session from results. `user_command_history()` (#104) aggregates the user's OWN successful `!` commands across recent sessions (only `decision:"user-direct"` records whose `cmd_end` reports exit 0 — model tool-loop commands never pollute the palette) and feeds the web terminal-mode autocomplete via the hello payload's `cmd_history`.
- **`prompt.py`** — the boxed input UI, built as a small prompt_toolkit `Application` (not `PromptSession`) because the footer-under-input layout requires it.
- **`server.py`** — `aish-web`: the same Agent behind a Starlette WebSocket instead of a TTY. `Bridge` is the core mechanism: `run_task` runs via `asyncio.to_thread`, callbacks emit JSON events through `call_soon_threadsafe`, and the approval callbacks block the worker on a `queue.Queue` slot until the browser answers — the approval gate is unchanged, only the transport differs. The process holds many open `Session`s (each its own Agent + SessionLog + Bridge + busy flag, capped at `MAX_OPEN_SESSIONS` with idle eviction) and serves many connections at once (**multi-connection, #102**): each socket is a `Client` (its own `ws`, `outbox`, `sender`, and independently-chosen `viewing` session), tracked in `WebServer.clients`; per-connection state that used to be singletons on `WebServer` (`self.ws`/`self.sender`/`self._active`) now lives on the Client. `Bridge._put` fans each event out to EVERY viewer's outbox (`Session.viewers`, owned by the bridge), and each Client's own sender coroutine drains its own outbox to its own socket — the transcript-append + all viewer-outbox pushes happen on the loop thread (via `call_soon_threadsafe`/`_show`), so no locking is needed (the invariant that kept replay snapshots race-free). Connections coexist WITHOUT preempting — `_attach` no longer closes the previous socket (the old single-client `CLOSE_REPLACED` path is gone); a bare connection (no `?session=`) lands on the default startup session (`WebServer._default`), a named one on `?session=`. Background sessions keep running, their events buffering into their transcript; switching a Client's view = hello + full transcript replay (`_show`), which is also what makes phone lock/unlock lossless. Control is **last-actor-drives**: every action handler calls `_claim(client)` first, stamping `Session.controller` and (when it changes) broadcasting a NON-recorded `role` event (`{controller, you}`) to that session's viewers; view messages (switch, files, jobs, dequeue, reconnect) never claim. There is no locked role and no disabled UI — the approval card fans out to all viewers, any may answer, and the asyncio loop serializes messages so exactly one `answer()` reaches the blocked worker (`Bridge.answer`'s pending-slot guard drops the duplicate). On disconnect (`_detach`→`_leave`) the Client leaves its session's viewers and, if it was controller, releases control (`role` → `controller:null`). `_evict_idle` never closes a busy session, one with viewers, or the default. The `role` event is never recorded, so cold replay re-derives control from live membership. The `active` property survives only as a representative-session helper for HTTP endpoints (no socket context) and tests; `/file` scopes to the union of all open sessions' roots. A session that fell out of memory (idle-evicted or after a restart) is reopened cold from its log and reconstructed into the same replay event stream (`SessionLog.reconstruct_events`), so its activity trace looks identical hot or cold. The vanilla-JS frontend lives in `aish/static/` (no build step, iOS-styled, inline SVG icons). Standalone PWA mode hides Safari's chrome, so a stalled socket has no browser refresh to fall back on (#18): `reconnect()` in `app.js` force-closes the current `ws` (even one still reporting `readyState OPEN` — a "zombie" socket the browser hasn't noticed died) and reconnects immediately, bypassing the backoff ladder. It's reachable from the session-title `⋯` menu ("Reconnect") and by tapping the existing `#connbar` "reconnecting…" bar itself. `requestNewChat()` wraps New Chat's `send({type:"new"})` to reconnect-then-retry when the socket is in a *detectable* dead state (CLOSING/CLOSED/CONNECTING) instead of silently no-op'ing; a zombie socket isn't detectable client-side, which is why the manual Reconnect control exists as the general fix. Pull-to-refresh was considered and skipped: `messagesEl` already runs a hand-rolled horizontal swipe-paging gesture with delicate vertical-vs-horizontal disambiguation (touchstart/touchmove/touchend near the bottom of `app.js`), and layering a vertical pull gesture on the same element risked fighting that logic for a case the menu/connbar controls already cover. Web approvers mirror `cli.make_approver` exactly (denylist first, also on edited commands; auto-approval scoped to live roots; per-session "allow this session" prefixes; "Always allow" saves the card's shown prefixes to the same persistent allowlist file as the CLI's `a` answer). **Mid-task steering (#95):** a `/cd` or a message typed while a task runs is applied/injected BETWEEN steps, not only after the task. The agent's step loop polls two get/drain callbacks at the top of each iteration (`check_pending_cwd`, `check_pending_messages`, wired from `open_session` over `session_holder`; lock-free — the event loop is the sole setter, the worker thread the sole consumer). `check_pending_cwd` get-and-clears `Session.pending_cwd` and the agent `rebase()`s + rebuilds `messages[0]` for the new dir; `check_pending_messages` drains the text-only items of `Session.queue` and injects each as a mid-task user turn (echo + a distinct `injected` trace step that renders live AND replays via `reconstruct_events` — the text is appended straight to `self.messages`, NOT logged as a conversation `message` record, so cold replay does not split the turn; trade-off: injected steering is not carried into `--resume` history). Queued items carrying attachments — and `!`-prefixed terminal commands (#105), which must run through the user-direct path, not as model prompts — stay in the queue and run as a normal follow-up at `_finish_turn` (the safety net for anything left, e.g. queued after the final step). Every cwd move — mid-task, immediate `/cd`, or the post-task drain — now flows through the single `on_state("cwd", …)` handler, which is the ONE place that emits the timeline `workspace` marker AND (record=False) retires the #92 queue card (`cwd_dequeued`) + refreshes the top-bar chip (`cwd_changed`); `_cd`/`_finish_turn`/`run_user_command` no longer emit those manually. claude-max runs its own loop, so mid-task polling does not apply there (its inner Agent's post-task `/cd` still flows through the unified path). **Live-output coalescing (#109):** `StreamCoalescer` batches run_command's per-line output into fewer `stream` events (flush at 50 lines / 16 KB / 0.1 s or command end) so huge output can't lock the tab with per-line DOM appends; it is LIVE-ONLY — logged `tool` output and cold replay are untouched, the frontend re-splits on `\n`, so hot/cold trace parity holds. **Feedback → GitHub issue (#110/#130), two flavours:** text-only web `/feedback` uses the BLOCK flow — the model emits the finished issue as one ` ```aish-issue ` fenced block (line 1 `title:`, optional `---`, body verbatim; parsed by `parse_issue_block`, mirrored in app.js `issueDraftCard`) and never runs `gh issue create`; the frontend renders it as a review card and on confirm the backend files it VERBATIM on the pinned `ISSUE_REPO` (user-direct action, no approval gate). CLI feedback, and web feedback carrying attachments, use the CLASSIC flow — the model drafts in rendered markdown and runs `gh issue create` itself through the approval gate (it must upload assets, which the backend path doesn't handle). Attachments are published to a PUBLIC release, so consent is explicit: the draft lists every file with a per-file exclude chip (`FEEDBACK_ASSETS_RULES`), and a block-flow draft that gains attachments mid-adjustment auto-switches to classic via `FEEDBACK_SWITCH_NOTE` (appended server-side, model-only). `parse_feedback` in `cli.py` takes `block_flow`/`attachments` flags; the CLI always passes neither. **PDF export** endpoints (`handle_export_answer` / whole-session) call `export.py` with `_image_roots()` — the union of open sessions' roots — as the local-image trust boundary. Approval-card accents are classified client-side (#107): only destructive-looking commands (`looks_destructive` in app.js — sudo/rm/mv/kill/--force) get the orange danger accent; everything else uses the calm blue — the gate itself is unchanged. Voice (#97) is frontend-only: mic dictation into the composer (SpeechRecognition, hold-to-switch language), auto read-aloud of replies to dictated messages (speechSynthesis, per-paragraph language detection), read-aloud controls docked in the composer row, and a `/mic` diagnostic — no server involvement.
- **`export.py`** — local Markdown → PDF for the web UI (#64): `markdown` → HTML → `xhtml2pdf` (pure Python, no system libs), the text never leaves the machine. Two shapes: one answer, or a whole session's FINAL answers — "final" is structural (an assistant message NOT immediately followed by a tool result), which cleanly drops thinking/tool turns. Media embedding (#133): local `![](path)` images inline as base64 only when the symlink-resolved path stays inside the caller's `image_roots` (same trust boundary as approval — anything outside becomes a captioned link card, never read); remote images, whitelisted Google static-map snapshots (needs `GOOGLE_MAPS_API_KEY`), and YouTube thumbnails ARE fetched at export time, each with a timeout + size cap and link-card fallback. Heavy imports are deferred into the render function so server startup stays cheap.
- **`dir_ignore.py`** — the configurable gitignore-style ignore list shared by the web folder browser and @-file completion (#87), user-editable via config.toml `[directory_picker] ignore` (defaults written back so they're visible, mirroring aliases.py; malformed config degrades to defaults, never an empty picker). Matching is deliberately name-level `fnmatch` on basenames (trailing `/` = dirs only) — a pure in-memory filter that must never add a per-subfolder stat (that caused the #86 freeze).
- **`skills.py`** — the knowledge store: skills (playbooks, `./.aish/skills/` project-wins or `~/.config/aish/skills/`) and memory (one fact per file, same md+frontmatter format, `.aish/memory/` dirs), plus legacy `lessons.md` lines as synthetic memory entries until `/learn lessons` migrates them. A skill is either a flat `<name>.md` OR an agentskills.io-compatible folder `<name>/SKILL.md` bundling `scripts/`/`references/`/`assets/` (name defaults to the dir, frontmatter still wins; `_dir_entries` scans subdirs for `SKILL.md` only for `kind=="skill"`, so memory stays flat); `read_skill`/`load_skill` append a `_bundled_note` naming the folder's files so the model reads/runs them via read_file/run_command. This is the substrate for the **skills-primary, tools-as-scalpel** capability model (epic #141): tools are added later only for hot, shell-fragile, reliability-critical ops that a documented skill snippet can't do safely. Progressive disclosure is the design rule: `knowledge_index()` renders a capped, byte-stable name+description index that `agent.compose_system_content()` rebuilds into `messages[0]` at EVERY `run_task` (that live rescan is what makes new skills appear without restart — don't reintroduce boot-time caching); full bodies load via `read_skill`; the long tail is searched by the `recall` tool (`rank_entries` + `recall_text`: deterministic difflib tiers, two-phase, hard caps, mtime-cached parsing). A per-task `TASK_REMINDER` system message is re-inserted before each user message (directly, never via `_append`, so it stays out of logs/transcripts) — recency is what makes small local models actually consult skills. `remember` writes via `save_memory` (slug-validated, create-or-update, deduped) and stays auto-approved; skill files are written with normal diff-approved `write_file`/`edit_file`. **Pre-flight injection (#40/#43):** `skills.preflight` picks the top entries for a task and `run_task` injects their bodies into the per-task reminder up-front (capped by the `PREFLIGHT_*` constants; oversized bodies get a teaser + a read gate armed until `read_skill` loads them) — embedding similarity selects entries when a `SemanticIndex` is wired, with exact name/keyword hits kept as a deterministic guarantee rail. Index caps are per-type (`INDEX_SKILLS_MAX`, `INDEX_MEMORY_MAX`), so one entry type growing large can never evict the others from the prompt.
- **`embeddings.py`** — the semantic layer over knowledge retrieval (#43): one local Ollama embedding model (`embeddinggemma` by default — multilingual because tasks arrive in Polish while entries are English; `AISH_EMBED_MODEL` overrides) scores task-vs-entry similarity regardless of which chat backend runs the task, so retrieval behaves identically on every `--model` and the corpus never leaves the machine. Entries embed as a single identity line (`name: description (keywords: …)`), never bodies — selection reads identity, so vectors stay stable while playbooks grow; vectors cache in the state dir keyed by sha256(model + text). Retrieval-tuned models need task-type prefixes Ollama does NOT add (`_PREFIXES`; skipping them measurably collapses similarity separation ~2x). Any failure (ollama down, model not pulled) makes `scores()` return `None` and callers fall back to the lexical tiers — lexical word-matching is the guaranteed floor, embeddings are an upgrade, never a dependency. Both entry points wire it the same way: `Agent(semantic=SemanticIndex(state_dir))`.
- **`tool_plugins.py`** — plugin tools (epic #141, the tools-as-scalpel half): droppable `TOOL.md` manifests the model calls as structured tool calls, **indistinguishable from native tools** because `to_tool_def()` emits the exact `{"type":"function","function":{…}}` shape `tools.TOOL_SCHEMAS` uses and `agent._dispatch` routes both through the same gate. A tool is a folder `<name>/TOOL.md` under `./.aish/tools/` (project) or `~/.config/aish/tools/` (global), bundling an optional wrapper. Doctrine: **schema on the way IN** (validated JSON args handed to the executable on **stdin — no shell**, so free-text args can't be mangled by quoting; this is the one irreducible win over a skill's shell-invoked script), **prose on the way OUT** (raw stdout+stderr + `[exit code: N]`, no output schema). Frontmatter is dependency-free line parsing (`schema` is a JSON object string); `mutating` is **required** (fail-closed — a manifest that doesn't declare it is invalid, never silently read-only) and a **floor never authority**. The linter (`_parse_tool`) is deterministic and skips invalid manifests with a warning rather than crashing discovery; `resolve_executable` only runs a bundled `./wrapper` inside the tool dir or a bare PATH binary (no absolute/escaping paths). The agent rescans only when the dirs' `signature()` (mtime set) moves — a mid-task manifest edit is picked up on the next step. **Approval:** read-only tools auto-run; **mutating tools are gated by `approve_tool`**, a fourth approval callback wired from both entry points (`cli.make_tool_approver` → y/N; `server.make_web_approvers`' `approve_tool` → an `approval_request` of `kind:"tool"` rendered by `app.js`'s `buildToolCard`, **reusing the command card verbatim**). `_dispatch_plugin_tool` mirrors `run_command`'s #81 verdict semantics exactly: `True` runs, `None`/`False` denies, `Denied(comment)` STOPs (arms the stop gate + appends feedback), `Approved(comment)` HOLDs (args not run — the model reworks and re-proposes, re-approved before running). Mutating tools are exposed to the model ONLY when a tool approver is wired; without one they stay hidden and fail closed in dispatch — a mutation never runs ungated. There is deliberately **no denylist/auto-approval on tools** (a mutating tool always prompts): the tool's safety is manifest review (`create_tool` + human diff-approval) plus this per-call gate. Parallelizing read-only plugin calls (like native `READ_ONLY_TOOLS`) is a deferred optimization — they currently dispatch sequentially. **Authoring (`create_tool`, #138):** a native, model-autonomous tool (in `tools.TOOL_SCHEMAS`) that writes a new `TOOL.md` + wrapper. Three guardrails the model cannot bypass: the WHEN test lives IMPERATIVELY in the tool description (create only when frequent + shell-fragile args + reliability; otherwise write a skill — never for read-only/simple/one-off); `Agent._create_tool` LINTS the drafted manifest in a temp dir and refuses to write on any error (structured feedback → correct-and-retry, so a broken tool can't land); and both files go through the normal `approve_write` diff gate (wrapper first — the executable, the critical thing to review — then manifest; a denial aborts without an orphan). The wrapper is chmod +x'd after commit and the plugin signature reset so the new tool is offered next step. `wrapper_lang` sets the file/shebang (sh|python); `scope` picks global (`~/.config/aish/tools`, backed up by the knowledge-git agent) or project.

Startup safety: launching either entry point from `$HOME` re-anchors the session to `~/aish` (`cli.default_workspace`, also used by `create_app`) so the home tree never becomes the auto-approval root; explicit `cwd` overrides are respected.

Web-only UI conventions the model is told about (in `web_usage_context`): quick-reply chips — `[Label](aish-reply://answer text)` links rendered as one-tap buttons by `app.js` — and the approval-card feedback field that produces `Denied(comment)`.

## Testing pattern

Tests script the model side instead of running one: `FakeChat` (see `tests/test_agent.py`) returns pre-canned responses shaped like the ollama library's, injected via `Agent(client_chat=...)`. Tool implementations are monkeypatched at the module level (`agent_module.tools.read_docs`, `agent_module.web.web_search`). Follow this pattern — tests must run with no model, no network, no real command execution. Frontend logic gets dependency-free Node checks in `tests/js/`: the REAL function is extracted from app.js by source markers and run in a `vm` context against a minimal fake DOM — the shipped code is tested, never a hand-copied duplicate. `tests/test_frontend_js.py` runs each script inside the pytest suite (skipped when node is absent).

## Documentation duplication

The README's user-facing docs are also summarized in the agent's own system prompt (`SYSTEM_PROMPT_TEMPLATE` in `agent.py` — aish answers questions about itself). When changing user-visible behavior, update both the README and the system prompt text.
